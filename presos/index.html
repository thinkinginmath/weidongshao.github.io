Stuby library - gRPC

  • Google RPC: Websocket over HTTP protocols.
  • Works across languages and platforms.
  • Scales to millions of RPCs easily provided by this platform.
  • http://www.grpc.io/
  • Built-in HTTP service on every server.
  • Simple app management and monitoring through http server
  • Bi-directional streaming with http/2 based transport
  • Used by many companies as their sole streaming service.

X As A Service

  • Cloud based common services for development and testing
  • Each service maintained separately, providing better scalability and lower costs for maintaining individual services for different teams

X As A Service - Examples

  • RDB: hosted MySQL
  • BigTable: Hosted services for each project for every data center
  • GFS/CNS: Centralized FS. Disable access to filesystem on each server, easy to manage when service failures occur.

X As A Service - Benefits

  • Rapid prototyping and app launch - reduce TTM
  • Reduce operational cost - pooled resources
  • Better performace - by domain-specific experts
  • cloud based file system provides high availability and failover (by encoding such as Reed–Solomon error correction)

gRPC and Common Services Benefits

gRPC and common services make high availablity, high scalibity and high performance software design much easier.

Decouple Software From Hardware

  • No special hardware-dependent code, making it more portable.
  • Unified platform and infrastructure with unified API, making it easier to manage.
  • Lower cost in operation when there’s no need to maintain hardware-specific code.

Decouple Software From Hardware

  • Example: Google Borg, the internal container management system
  • Borg manages the resource pool so that server utilitization is greatly increased.
  • Significantly reduce CapEx and OpEx (esp. in large scale operation)

Decouple Software From Hardware

  • some small portition of code that needs high performance hardware can be deployed on the dedicated central-managed hardwares
    • using rpc to communicate to components.
    • CPU with AVX2 capability,
    • GPU
    • TPU (for google machine learning tensor framework https://en.wikipedia.org/wiki/Tensor_processing_unit))

Decouple Application from Infrastructure

  • Application level SREs don’t need to know the details of Borg internals.
  • This type of platform-infrastructure design drives the software engineers to write programs in a better way.
  • New design for H/A, Load Balancing, Data Persistence and so on.
  • Natural need for microservice architecture
